			<h1 id="firstHeading" class="firstHeading" lang="en">MapReduce</h1>
						<div id="bodyContent" class="mw-body-content">
									<div id="siteSub">From Wikipedia, the free encyclopedia</div>
								<div id="contentSub"></div>
												<div id="jump-to-nav" class="mw-jump">
					Jump to:					<a href="https://en.wikipedia.org/wiki/MapReduce#mw-head">navigation</a>, 					<a href="https://en.wikipedia.org/wiki/MapReduce#p-search">search</a>
				</div>
				<div id="mw-content-text" lang="en" dir="ltr" class="mw-content-ltr"><p><b>MapReduce</b> is a <a href="https://en.wikipedia.org/wiki/Programming_model" title="Programming model" class="mw-redirect">programming model</a> and an associated implementation for processing and generating large data sets with a <a href="https://en.wikipedia.org/wiki/Parallel_computing" title="Parallel computing">parallel</a>, <a href="https://en.wikipedia.org/wiki/Distributed_computing" title="Distributed computing">distributed</a> algorithm on a <a href="https://en.wikipedia.org/wiki/Cluster_(computing)" title="Cluster (computing)" class="mw-redirect">cluster</a>.<sup id="cite_ref-1" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-1"><span>[</span>1<span>]</span></a></sup><sup id="cite_ref-2" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-2"><span>[</span>2<span>]</span></a></sup> Conceptually similar approaches have been very well known since 1995 with the <a href="https://en.wikipedia.org/wiki/Message_Passing_Interface" title="Message Passing Interface">Message Passing Interface</a> <sup id="cite_ref-3" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-3"><span>[</span>3<span>]</span></a></sup> standard having reduce <sup id="cite_ref-4" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-4"><span>[</span>4<span>]</span></a></sup> and scatter operations.<sup id="cite_ref-5" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-5"><span>[</span>5<span>]</span></a></sup></p>
<p>A MapReduce program is composed of a <a href="https://en.wikipedia.org/wiki/Map_(parallel_pattern)" title="Map (parallel pattern)"><b>Map()</b></a> <a href="https://en.wikipedia.org/wiki/Procedure_(computing)" title="Procedure (computing)" class="mw-redirect">procedure</a> that performs filtering and sorting (such as sorting students by first name into queues, one queue for each name) and a <b>Reduce()</b> procedure that performs a summary operation (such as counting the number of students in each queue, yielding name frequencies). The "MapReduce System" (also called "infrastructure" or "framework") orchestrates the processing by <a href="https://en.wikipedia.org/wiki/Marshalling_(computer_science)" title="Marshalling (computer science)">marshalling</a> the distributed servers, running the various tasks in parallel, managing all communications and data transfers between the various parts of the system, and providing for <a href="https://en.wikipedia.org/wiki/Redundancy_(engineering)" title="Redundancy (engineering)">redundancy</a> and <a href="https://en.wikipedia.org/wiki/Fault-tolerant_computer_system" title="Fault-tolerant computer system">fault tolerance</a>.</p>
<p>The model is inspired by the <a href="https://en.wikipedia.org/wiki/Map_(higher-order_function)" title="Map (higher-order function)">map</a> and <a href="https://en.wikipedia.org/wiki/Fold_(higher-order_function)" title="Fold (higher-order function)">reduce</a> functions commonly used in <a href="https://en.wikipedia.org/wiki/Functional_programming" title="Functional programming">functional programming</a>,<sup id="cite_ref-map_6-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-map-6"><span>[</span>6<span>]</span></a></sup> although their purpose in the MapReduce framework is not the same as in their original forms.<sup id="cite_ref-7" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-7"><span>[</span>7<span>]</span></a></sup> The key contributions of the MapReduce framework are not the actual map and reduce functions, but the scalability and fault-tolerance achieved for a variety of applications by optimizing the execution engine once. As such, a <a href="https://en.wikipedia.org/wiki/Single-threaded" title="Single-threaded" class="mw-redirect">single-threaded</a> implementation of MapReduce (such as <a href="https://en.wikipedia.org/wiki/MongoDB" title="MongoDB">MongoDB</a>) will usually not be faster than a traditional (non-MapReduce) implementation, any gains are usually only seen with <a href="https://en.wikipedia.org/wiki/Multi-threaded" title="Multi-threaded" class="mw-redirect">multi-threaded</a> implementations.<sup id="cite_ref-stackoverflow_8-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-stackoverflow-8"><span>[</span>8<span>]</span></a></sup> The use of this model is beneficial only when the optimized distributed shuffle operation (which reduces network communication cost) and fault tolerance features of the MapReduce framework come into play. Optimizing the communication cost is essential to a good MapReduce algorithm.<sup id="cite_ref-ullman_9-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-ullman-9"><span>[</span>9<span>]</span></a></sup></p>
<p>MapReduce <a href="https://en.wikipedia.org/wiki/Library_(software)" title="Library (software)" class="mw-redirect">libraries</a> have been written in many programming languages, with different levels of optimization. A popular <a href="https://en.wikipedia.org/wiki/Open-source_software" title="Open-source software">open-source</a> implementation that has support for distributed shuffles is part of <a href="https://en.wikipedia.org/wiki/Apache_Hadoop" title="Apache Hadoop">Apache Hadoop</a>. The name MapReduce originally referred to the proprietary <a href="https://en.wikipedia.org/wiki/Google" title="Google">Google</a> technology, but has since been <a href="https://en.wikipedia.org/wiki/Generic_trademark" title="Generic trademark">genericized</a>.</p>

<h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="https://en.wikipedia.org/w/index.php?title=MapReduce&action=edit&section=1" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>MapReduce is a framework for processing <a href="https://en.wikipedia.org/wiki/Parallel_computing" title="Parallel computing">parallelizable</a> problems across huge datasets using a large number of computers (nodes), collectively referred to as a <a href="https://en.wikipedia.org/wiki/Computer_cluster" title="Computer cluster">cluster</a> (if all nodes are on the same local network and use similar hardware) or a <a href="https://en.wikipedia.org/wiki/Grid_Computing" title="Grid Computing" class="mw-redirect">grid</a> (if the nodes are shared across geographically and administratively distributed systems, and use more heterogenous hardware). Processing can occur on data stored either in a <a href="https://en.wikipedia.org/wiki/Filesystem" title="Filesystem" class="mw-redirect">filesystem</a> (unstructured) or in a <a href="https://en.wikipedia.org/wiki/Database" title="Database">database</a> (structured). MapReduce can take advantage of locality of data, processing it on or near the storage assets in order to reduce the distance over which it must be transmitted.</p>
<ul>
<li><b>"Map" step:</b> Each worker node applies the "map()" function to the local data, and writes the output to a temporary storage. A master node orchestrates that for redundant copies of input data, only one is processed.</li>
<li><b>"Shuffle" step:</b> Worker nodes redistribute data based on the output keys (produced by the "map()" function), such that all data belonging to one key is located on the same worker node.</li>
<li><b>"Reduce" step:</b> Worker nodes now process each group of output data, per key, in parallel.</li>
</ul>
<p>MapReduce allows for distributed processing of the map and reduction operations. Provided that each mapping operation is independent of the others, all maps can be performed in parallel&nbsp;– though in practice this is limited by the number of independent data sources and/or the number of CPUs near each source. Similarly, a set of 'reducers' can perform the reduction phase, provided that all outputs of the map operation that share the same key are presented to the same reducer at the same time, or that the reduction function is <a href="https://en.wikipedia.org/wiki/Associative_property" title="Associative property">associative</a>. While this process can often appear inefficient compared to algorithms that are more sequential, MapReduce can be applied to significantly larger datasets than "commodity" servers can handle&nbsp;– a large <a href="https://en.wikipedia.org/wiki/Server_farm" title="Server farm">server farm</a> can use MapReduce to sort a <a href="https://en.wikipedia.org/wiki/Petabyte" title="Petabyte">petabyte</a> of data in only a few hours.<sup id="cite_ref-10" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-10"><span>[</span>10<span>]</span></a></sup> The parallelism also offers some possibility of recovering from partial failure of servers or storage during the operation: if one mapper or reducer fails, the work can be rescheduled&nbsp;– assuming the input data is still available.</p>
<p>Another way to look at MapReduce is as a 5-step parallel and distributed computation:</p>
<ol>
<li><b>Prepare the Map() input</b> – the "MapReduce system" designates Map processors, assigns the input key value <i>K1</i> that each processor would work on, and provides that processor with all the input data associated with that key value.</li>
<li><b>Run the user-provided Map() code</b> – Map() is run exactly once for each <i>K1</i> key value, generating output organized by key values <i>K2</i>.</li>
<li><b>"Shuffle" the Map output to the Reduce processors</b> – the MapReduce system designates Reduce processors, assigns the <i>K2</i> key value each processor should work on, and provides that processor with all the Map-generated data associated with that key value.</li>
<li><b>Run the user-provided Reduce() code</b> – Reduce() is run exactly once for each <i>K2</i> key value produced by the Map step.</li>
<li><b>Produce the final output</b> – the MapReduce system collects all the Reduce output, and sorts it by <i>K2</i> to produce the final outcome.</li>
</ol>
<p>These five steps can be Logically thought of as running in sequence – each step starts only after the previous step is completed – although in practice they can be interleaved as long as the final result is not affected.</p>
<p>In many situations, the input data might already be distributed (<a href="https://en.wikipedia.org/wiki/Shard_(database_architecture)" title="Shard (database architecture)">"sharded"</a>) among many different servers, in which case step 1 could sometimes be greatly simplified by assigning Map servers that would process the locally present input data. Similarly, step 3 could sometimes be sped up by assigning Reduce processors that are as close as possible to the Map-generated data they need to process.</p>
<h2><span class="mw-headline" id="Logical_view">Logical view</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="https://en.wikipedia.org/w/index.php?title=MapReduce&action=edit&section=2" title="Edit section: Logical view">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The <i>Map</i> and <i>Reduce</i> functions of <i>MapReduce</i> are both defined with respect to data structured in (key, value) pairs. <i>Map</i> takes one pair of data with a type in one <a href="https://en.wikipedia.org/wiki/Data_domain" title="Data domain">data domain</a>, and returns a list of pairs in a different domain:</p>
<p><code>Map(k1,v1)</code> → <code>list(k2,v2)</code></p>
<p>The <i>Map</i> function is applied in parallel to every pair in the input dataset. This produces a list of pairs for each call. After that, the MapReduce framework collects all pairs with the same key from all lists and groups them together, creating one group for each key.</p>
<p>The <i>Reduce</i> function is then applied in parallel to each group, which in turn produces a collection of values in the same domain:</p>
<p><code>Reduce(k2, list (v2))</code> → <code>list(v3)</code></p>
<p>Each <i>Reduce</i> call typically produces either one value v3 or an empty return, though one call is allowed to return more than one value. The returns of all calls are collected as the desired result list.</p>
<p>Thus the MapReduce framework transforms a list of (key, value) pairs into a list of values. This behavior is different from the typical functional programming map and reduce combination, which accepts a list of arbitrary values and returns one single value that combines <i>all</i> the values returned by map.</p>
<p>It is <a href="https://en.wikipedia.org/wiki/Necessity_and_sufficiency" title="Necessity and sufficiency">necessary but not sufficient</a> to have implementations of the map and reduce abstractions in order to implement MapReduce. Distributed implementations of MapReduce require a means of connecting the processes performing the Map and Reduce phases. This may be a <a href="https://en.wikipedia.org/wiki/Distributed_file_system" title="Distributed file system" class="mw-redirect">distributed file system</a>. Other options are possible, such as direct streaming from mappers to reducers, or for the mapping processors to serve up their results to reducers that query them.</p>
<h3><span class="mw-headline" id="Examples">Examples</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="https://en.wikipedia.org/w/index.php?title=MapReduce&action=edit&section=3" title="Edit section: Examples">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The prototypical MapReduce example counts the appearance of each word in a set of documents:<sup id="cite_ref-11" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-11"><span>[</span>11<span>]</span></a></sup></p>
<pre><b>function</b> <u>map</u>(String name, String document):
  <i>// name: document name</i>
  <i>// document: document contents</i>
  <b>for each</b> word w <b>in</b> document:
    emit (w, 1)

<b>function</b> <u>reduce</u>(String word, Iterator partialCounts):
  <i>// word: a word</i>
  <i>// partialCounts: a list of aggregated partial counts</i>
  sum = 0
  <b>for each</b> pc <b>in</b> partialCounts:
    sum += ParseInt(pc)
  emit (word, sum)
</pre>
<p>Here, each document is split into words, and each word is counted by the <i>map</i> function, using the word as the result key. The framework puts together all the pairs with the same key and feeds them to the same call to <i>reduce</i>. Thus, this function just needs to sum all of its input values to find the total appearances of that word.</p>
<p>As another example, imagine that for a database of 1.1 billion people, one would like to compute the average number of social contacts a person has according to age. In <a href="https://en.wikipedia.org/wiki/SQL" title="SQL">SQL</a>, such a query could be expressed as:</p>
<div dir="ltr" class="mw-geshi mw-code mw-content-ltr">
<div class="sql source-sql">
<pre class="de1">  <span class="kw1">SELECT</span> age<span class="sy0">,</span> AVG<span class="br0">(</span>contacts<span class="br0">)</span>
    <span class="kw1">FROM</span> social<span class="sy0">.</span>person
<span class="kw1">GROUP</span> <span class="kw1">BY</span> age
<span class="kw1">ORDER</span> <span class="kw1">BY</span> age
</pre></div>
</div>
<p>Using MapReduce, the <tt>K1</tt> key values could be the integers 1 through 1100, each representing a batch of 1 million records, the <tt>K2</tt> key value could be a person’s age in years, and this computation could be achieved using the following functions:</p>
<pre><b>function</b> Map <b>is</b>
    <b>input:</b> <b>integer</b> K1 between 1 and 1100, representing a batch of 1 million social.person records
    <b>for each</b> social.person record in the K1 batch <b>do</b>
        <b>let</b> Y be the person's age
        <b>let</b> N be the number of contacts the person has
        <b>produce one output record</b> (Y,(N,1))
    <b>repeat</b>
<b>end function</b>

<b>function</b> Reduce <b>is</b>
    <b>input:</b> age (in years) Y
    <b>for each</b> input record (Y,(N,C)) <b>do</b>
        Accumulate in S the sum of N*C
        Accumulate in C<sub>new</sub> the sum of C
    <b>repeat</b>
    <b>let</b> A be S/C<sub>new</sub>
    <b>produce one output record</b> (Y,(A,C<sub>new</sub>))
<b>end function</b>
</pre>
<p>The MapReduce System would line up the 1100 Map processors, and would provide each with its corresponding 1 million input records. The Map step would produce 1.1 billion <tt>(Y,(N,1))</tt> records, with <tt>Y</tt> values ranging between, say, 8 and 103. The MapReduce System would then line up the 96 Reduce processors by performing shuffling operation of the key/value pairs due to the fact that we need average per age, and provide each with its millions of corresponding input records. The Reduce step would result in the much reduced set of only 96 output records <tt>(Y,A)</tt>, which would be put in the final result file, sorted by <tt>Y</tt>.</p>
<p>The count info in the record is important if the processing is reduced more than one time. If we did not add the count of the records, the computed average would be wrong, for example:</p>
<pre><i>-- map output #1: age, quantity of contacts</i>
10, 9
10, 9
10, 9
</pre>
<pre><i>-- map output #2: age, quantity of contacts</i>
10, 9
10, 9
</pre>
<pre><i>-- map output #3: age, quantity of contacts</i>
10, 10
</pre>
<p>If we reduce files <tt>#1</tt> and <tt>#2</tt>, we will have a new file with an average of 9 contacts for a 10 year old person ((9+9+9+9+9)/5):</p>
<pre><i>-- reduce step #1: age, average of contacts</i>
10, 9
</pre>
<p>If we reduce it with file <tt>#3</tt>, we lose the count of how many records we've already seen, so we end up with an average of 9.5 contacts for a 10 year old person ((9+10)/2), which is wrong. The correct answer is 9.17 ((9+9+9+9+9+10)/6).</p>
<h2><span class="mw-headline" id="Dataflow">Dataflow</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="https://en.wikipedia.org/w/index.php?title=MapReduce&action=edit&section=4" title="Edit section: Dataflow">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The frozen part of the MapReduce framework is a large distributed sort. The hot spots, which the application defines, are:</p>
<ul>
<li>an <i>input reader</i></li>
<li>a <i>Map</i> function</li>
<li>a <i>partition</i> function</li>
<li>a <i>compare</i> function</li>
<li>a <i>Reduce</i> function</li>
<li>an <i>output writer</i></li>
</ul>
<h3><span class="mw-headline" id="Input_reader">Input reader</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="https://en.wikipedia.org/w/index.php?title=MapReduce&action=edit&section=5" title="Edit section: Input reader">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The <i>input reader</i> divides the input into appropriate size 'splits' (in practice typically 64&nbsp;MB to 128&nbsp;MB) and the framework assigns one split to each <i>Map</i> function. The <i>input reader</i> reads data from stable storage (typically a <a href="https://en.wikipedia.org/wiki/Distributed_file_system" title="Distributed file system" class="mw-redirect">distributed file system</a>) and generates key/value pairs.</p>
<p>A common example will read a directory full of text files and return each line as a record.</p>
<h3><span class="mw-headline" id="Map_function">Map function</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="https://en.wikipedia.org/w/index.php?title=MapReduce&action=edit&section=6" title="Edit section: Map function">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The <i>Map</i> function takes a series of key/value pairs, processes each, and generates zero or more output key/value pairs. The input and output types of the map can be (and often are) different from each other.</p>
<p>If the application is doing a word count, the map function would break the line into words and output a key/value pair for each word. Each output pair would contain the word as the key and the number of instances of that word in the line as the value.</p>
<h3><span class="mw-headline" id="Partition_function">Partition function</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="https://en.wikipedia.org/w/index.php?title=MapReduce&action=edit&section=7" title="Edit section: Partition function">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Each <i>Map</i> function output is allocated to a particular <i>reducer</i> by the application's <i>partition</i> function for <a href="https://en.wikipedia.org/wiki/Sharding" title="Sharding" class="mw-redirect">sharding</a> purposes. The <i>partition</i> function is given the key and the number of reducers and returns the index of the desired <i>reducer</i>.</p>
<p>A typical default is to <a href="https://en.wikipedia.org/wiki/Hash_function" title="Hash function">hash</a> the key and use the hash value <a href="https://en.wikipedia.org/wiki/Modulo_operation" title="Modulo operation">modulo</a> the number of <i>reducers</i>. It is important to pick a partition function that gives an approximately uniform distribution of data per shard for <a href="https://en.wikipedia.org/wiki/Load_balancing_(computing)" title="Load balancing (computing)">load-balancing</a> purposes, otherwise the MapReduce operation can be held up waiting for slow reducers (reducers assigned more than their share of data) to finish.</p>
<p>Between the map and reduce stages, the data is <i>shuffled</i> (parallel-sorted / exchanged between nodes) in order to move the data from the map node that produced it to the shard in which it will be reduced. The shuffle can sometimes take longer than the computation time depending on network bandwidth, CPU speeds, data produced and time taken by map and reduce computations.</p>
<h3><span class="mw-headline" id="Comparison_function">Comparison function</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="https://en.wikipedia.org/w/index.php?title=MapReduce&action=edit&section=8" title="Edit section: Comparison function">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The input for each <i>Reduce</i> is pulled from the machine where the <i>Map</i> ran and sorted using the application's <i>comparison</i> function.</p>
<h3><span class="mw-headline" id="Reduce_function">Reduce function</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="https://en.wikipedia.org/w/index.php?title=MapReduce&action=edit&section=9" title="Edit section: Reduce function">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The framework calls the application's <i>Reduce</i> function once for each unique key in the sorted order. The <i>Reduce</i> can iterate through the values that are associated with that key and produce zero or more outputs.</p>
<p>In the word count example, the <i>Reduce</i> function takes the input values, sums them and generates a single output of the word and the final sum.</p>
<h3><span class="mw-headline" id="Output_writer">Output writer</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="https://en.wikipedia.org/w/index.php?title=MapReduce&action=edit&section=10" title="Edit section: Output writer">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The <i>Output Writer</i> writes the output of the <i>Reduce</i> to the stable storage.</p>
<h2><span class="mw-headline" id="Performance_considerations">Performance considerations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="https://en.wikipedia.org/w/index.php?title=MapReduce&action=edit&section=11" title="Edit section: Performance considerations">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>MapReduce programs are not guaranteed to be fast. The main benefit of this programming model is to exploit the optimized shuffle operation of the platform, and only having to write the <i>Map</i> and <i>Reduce</i> parts of the program. In practice, the author of a MapReduce program however has to take the shuffle step into consideration; in particular the partition function and the amount of data written by the <i>Map</i> function can have a large impact on the performance. Additional modules such as the <i>Combiner</i> function can help to reduce the amount of data written to disk, and transmitted over the network.</p>
<p>When designing a MapReduce algorithm, the author needs to choose a good tradeoff<sup id="cite_ref-ullman_9-1" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-ullman-9"><span>[</span>9<span>]</span></a></sup> between the computation and the communication costs. Communication cost often dominates the computation cost,<sup id="cite_ref-ullman_9-2" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-ullman-9"><span>[</span>9<span>]</span></a></sup> and many MapReduce implementations are designed to write all communication to distributed storage for crash recovery.</p>
<p>For processes that complete fast, and where the data fits into main memory of a single machine or a small cluster, using a MapReduce framework usually is not effective: since these frameworks are designed to recover from the loss of whole nodes during the computation, they write interim results to distributed storage. This crash recovery is expensive, and only pays off when the computation involves many computers and a long runtime of the computation - a task that completes in seconds can just be restarted in the case of an error; and the likelihood of at least one machine failing grows quickly with the cluster size. On such problems, implementations keeping all data in memory and simply restarting a computation on node failures, or - when the data is small enough - non-distributed solutions will often be faster than a MapReduce system.</p>
<h2><span class="mw-headline" id="Distribution_and_reliability">Distribution and reliability</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="https://en.wikipedia.org/w/index.php?title=MapReduce&action=edit&section=12" title="Edit section: Distribution and reliability">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>MapReduce achieves reliability by parceling out a number of operations on the set of data to each node in the network. Each node is expected to report back periodically with completed work and status updates. If a node falls silent for longer than that interval, the master node (similar to the master server in the <a href="https://en.wikipedia.org/wiki/Google_File_System" title="Google File System">Google File System</a>) records the node as dead and sends out the node's assigned work to other nodes. Individual operations use <a href="https://en.wikipedia.org/wiki/Atomicity_(programming)" title="Atomicity (programming)" class="mw-redirect">atomic</a> operations for naming file outputs as a check to ensure that there are not parallel conflicting threads running. When files are renamed, it is possible to also copy them to another name in addition to the name of the task (allowing for <a href="https://en.wikipedia.org/wiki/Side-effect_(computer_science)" title="Side-effect (computer science)" class="mw-redirect">side-effects</a>).</p>
<p>The reduce operations operate much the same way. Because of their inferior properties with regard to parallel operations, the master node attempts to schedule reduce operations on the same node, or in the same rack as the node holding the data being operated on. This property is desirable as it conserves bandwidth across the backbone network of the datacenter.</p>
<p>Implementations are not necessarily highly reliable. For example, in older versions of <a href="https://en.wikipedia.org/wiki/Hadoop" title="Hadoop" class="mw-redirect">Hadoop</a> the <i>NameNode</i> was a <a href="https://en.wikipedia.org/wiki/Single_point_of_failure" title="Single point of failure">single point of failure</a> for the distributed filesystem. Later versions of Hadoop have high availability with an active/passive failover for the "NameNode."</p>
<h2><span class="mw-headline" id="Uses">Uses</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="https://en.wikipedia.org/w/index.php?title=MapReduce&action=edit&section=13" title="Edit section: Uses">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>MapReduce is useful in a wide range of applications, including distributed pattern-based searching, distributed sorting, web link-graph reversal, Singular Value Decomposition,<sup id="cite_ref-12" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-12"><span>[</span>12<span>]</span></a></sup> web access log stats, <a href="https://en.wikipedia.org/wiki/Inverted_index" title="Inverted index">inverted index</a> construction, <a href="https://en.wikipedia.org/wiki/Document_clustering" title="Document clustering">document clustering</a>, <a href="https://en.wikipedia.org/wiki/Machine_learning" title="Machine learning">machine learning</a>,<sup id="cite_ref-mrml_13-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-mrml-13"><span>[</span>13<span>]</span></a></sup> and <a href="https://en.wikipedia.org/wiki/Statistical_machine_translation" title="Statistical machine translation">statistical machine translation</a>. Moreover, the MapReduce model has been adapted to several computing environments like multi-core and many-core systems,<sup id="cite_ref-evalMR_14-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-evalMR-14"><span>[</span>14<span>]</span></a></sup><sup id="cite_ref-graphicsMR_15-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-graphicsMR-15"><span>[</span>15<span>]</span></a></sup><sup id="cite_ref-tiledMR_16-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-tiledMR-16"><span>[</span>16<span>]</span></a></sup> desktop grids,<sup id="cite_ref-gridMR_17-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-gridMR-17"><span>[</span>17<span>]</span></a></sup> volunteer computing environments,<sup id="cite_ref-volunteerMR_18-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-volunteerMR-18"><span>[</span>18<span>]</span></a></sup> dynamic cloud environments,<sup id="cite_ref-dynCloudMR_19-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-dynCloudMR-19"><span>[</span>19<span>]</span></a></sup> and mobile environments.<sup id="cite_ref-mobileMR_20-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-mobileMR-20"><span>[</span>20<span>]</span></a></sup></p>
<p>At Google, MapReduce was used to completely regenerate Google's index of the <a href="https://en.wikipedia.org/wiki/World_Wide_Web" title="World Wide Web">World Wide Web</a>. It replaced the old <i>ad hoc</i> programs that updated the index and ran the various analyses.<sup id="cite_ref-usage_21-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-usage-21"><span>[</span>21<span>]</span></a></sup> Development at Google has since moved on to technologies such as Percolator, Flume and MillWheel that offer streaming operation and updates instead of batch processing, to allow integrating "live" search results without rebuilding the complete index.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="https://en.wikipedia.org/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (February 2015)">citation needed</span></a></i>]</sup></p>
<p>MapReduce's stable inputs and outputs are usually stored in a <a href="https://en.wikipedia.org/wiki/Distributed_file_system" title="Distributed file system" class="mw-redirect">distributed file system</a>. The transient data is usually stored on local disk and fetched remotely by the reducers.</p>
<h2><span class="mw-headline" id="Criticism">Criticism</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="https://en.wikipedia.org/w/index.php?title=MapReduce&action=edit&section=14" title="Edit section: Criticism">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Lack_of_novelty">Lack of novelty</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="https://en.wikipedia.org/w/index.php?title=MapReduce&action=edit&section=15" title="Edit section: Lack of novelty">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="https://en.wikipedia.org/wiki/David_DeWitt" title="David DeWitt">David DeWitt</a> and <a href="https://en.wikipedia.org/wiki/Michael_Stonebraker" title="Michael Stonebraker">Michael Stonebraker</a>, computer scientists specializing in <a href="https://en.wikipedia.org/wiki/Parallel_database" title="Parallel database">parallel databases</a> and <a href="https://en.wikipedia.org/wiki/Shared-nothing_architecture" title="Shared-nothing architecture" class="mw-redirect">shared-nothing architectures</a>, have been critical of the breadth of problems that MapReduce can be used for.<sup id="cite_ref-shark_22-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-shark-22"><span>[</span>22<span>]</span></a></sup> They called its interface too low-level and questioned whether it really represents the <a href="https://en.wikipedia.org/wiki/Paradigm_shift" title="Paradigm shift">paradigm shift</a> its proponents have claimed it is.<sup id="cite_ref-ddandms1_23-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-ddandms1-23"><span>[</span>23<span>]</span></a></sup> They challenged the MapReduce proponents' claims of novelty, citing <a href="https://en.wikipedia.org/wiki/Teradata" title="Teradata">Teradata</a> as an example of <a href="https://en.wikipedia.org/wiki/Prior_art" title="Prior art">prior art</a> that has existed for over two decades. They also compared MapReduce programmers to <a href="https://en.wikipedia.org/wiki/CODASYL" title="CODASYL">Codasyl</a> programmers, noting both are "writing in a <a href="https://en.wikipedia.org/wiki/Low-level_programming_language" title="Low-level programming language">low-level language</a> performing low-level record manipulation."<sup id="cite_ref-ddandms1_23-1" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-ddandms1-23"><span>[</span>23<span>]</span></a></sup> MapReduce's use of input files and lack of <a href="https://en.wikipedia.org/wiki/Logical_schema" title="Logical schema">schema</a> support prevents the performance improvements enabled by common database system features such as <a href="https://en.wikipedia.org/wiki/B-tree" title="B-tree">B-trees</a> and <a href="https://en.wikipedia.org/wiki/Partition_(database)" title="Partition (database)">hash partitioning</a>, though projects such as <a href="https://en.wikipedia.org/wiki/Pig_(programming_language)" title="Pig (programming language)" class="mw-redirect">Pig (or PigLatin)</a>, <a href="https://en.wikipedia.org/wiki/Sawzall_(programming_language)" title="Sawzall (programming language)">Sawzall</a>, <a href="https://en.wikipedia.org/wiki/Apache_Hive" title="Apache Hive">Apache Hive</a>,<sup id="cite_ref-ApacheHiveWiki_24-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-ApacheHiveWiki-24"><span>[</span>24<span>]</span></a></sup> <a rel="nofollow" class="external text" href="http://ysmart.cse.ohio-state.edu/">YSmart</a>,<sup id="cite_ref-YSmartPaper_25-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-YSmartPaper-25"><span>[</span>25<span>]</span></a></sup> <a href="https://en.wikipedia.org/wiki/HBase" title="HBase" class="mw-redirect">HBase</a><sup id="cite_ref-HBase_26-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-HBase-26"><span>[</span>26<span>]</span></a></sup> and <a href="https://en.wikipedia.org/wiki/BigTable" title="BigTable">BigTable</a><sup id="cite_ref-HBase_26-1" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-HBase-26"><span>[</span>26<span>]</span></a></sup><sup id="cite_ref-BigTablePaper_27-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-BigTablePaper-27"><span>[</span>27<span>]</span></a></sup> are addressing some of these problems.</p>
<p>Greg Jorgensen wrote an article rejecting these views.<sup id="cite_ref-gj1_28-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-gj1-28"><span>[</span>28<span>]</span></a></sup> Jorgensen asserts that DeWitt and Stonebraker's entire analysis is groundless as MapReduce was never designed nor intended to be used as a database.</p>
<p>DeWitt and Stonebraker have subsequently published a detailed benchmark study in 2009 comparing performance of <a href="https://en.wikipedia.org/wiki/Hadoop" title="Hadoop" class="mw-redirect">Hadoop's</a> MapReduce and <a href="https://en.wikipedia.org/wiki/RDBMS" title="RDBMS" class="mw-redirect">RDBMS</a> approaches on several specific problems.<sup id="cite_ref-sigmod_29-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-sigmod-29"><span>[</span>29<span>]</span></a></sup> They concluded that relational databases offer real advantages for many kinds of data use, especially on complex processing or where the data is used across an enterprise, but that MapReduce may be easier for users to adopt for simple or one-time processing tasks.</p>
<p>Google has been granted a patent on MapReduce.<sup id="cite_ref-patent_30-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-patent-30"><span>[</span>30<span>]</span></a></sup> However, there have been claims that this patent should not have been granted because MapReduce is too similar to existing products. For example, map and reduce functionality can be very easily implemented in <a href="https://en.wikipedia.org/wiki/Oracle_database" title="Oracle database" class="mw-redirect">Oracle's</a> <a href="https://en.wikipedia.org/wiki/PL/SQL" title="PL/SQL">PL/SQL</a> database oriented language<sup id="cite_ref-Curt_Monash_31-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-Curt_Monash-31"><span>[</span>31<span>]</span></a></sup> or is supported for developers transparently in distributed database architectures such as <a href="https://en.wikipedia.org/wiki/Clusterpoint" title="Clusterpoint">Clusterpoint</a> XML database<sup id="cite_ref-Clusterpoint_32-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-Clusterpoint-32"><span>[</span>32<span>]</span></a></sup> or <a href="https://en.wikipedia.org/wiki/MongoDB" title="MongoDB">MongoDB</a> NoSQL database.<sup id="cite_ref-MongoDB_33-0" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-MongoDB-33"><span>[</span>33<span>]</span></a></sup></p>
<h3><span class="mw-headline" id="Restricted_programming_framework">Restricted programming framework</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="https://en.wikipedia.org/w/index.php?title=MapReduce&action=edit&section=16" title="Edit section: Restricted programming framework">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>MapReduce tasks must be written as acyclic dataflow programs, i.e. a stateless mapper followed by a stateless reducer, that are executed by a batch job scheduler. This paradigm makes repeated querying of datasets difficult and imposes limitations that are felt in fields such as <a href="https://en.wikipedia.org/wiki/Machine_learning" title="Machine learning">machine learning</a>, where iterative algorithms that revisit a single <a href="https://en.wikipedia.org/wiki/Working_set" title="Working set">working set</a> multiple times are the norm.<sup id="cite_ref-34" class="reference"><a href="https://en.wikipedia.org/wiki/MapReduce#cite_note-34"><span>[</span>34<span>]</span></a></sup></p>
